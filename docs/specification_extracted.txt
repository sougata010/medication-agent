1. Executive Summary
MedGraph AI is a production-grade, multi-agent medication intelligence platform built on LangGraph. It is designed to help patients understand their prescriptions, check for drug interactions, receive timely medication reminders, and manage their medication history — all in one place.
The platform deliberately avoids diagnosing diseases or recommending treatments. It is a medication companion, not a medical practitioner. Every answer is grounded in pharmacological data, medical guidelines, and the user's personal medication history — never in speculation.
2. System Architecture Overview
The platform is organized into four logical layers that separate concerns cleanly and allow each layer to scale independently.
Figure 1 — High-level system architecture of MedGraph AI
2.1  Layer Breakdown
2.2  Core Design Principles
Modularity: each agent is an isolated, replaceable node in the graph.
Parallelism: independent subtasks (medicine API, interaction check, image search) run concurrently via LangGraph fan-out.
Explainability: every response traces back to a specific data source or API call — no opaque black-box answers.
Safety-first: a dedicated Safety Layer intercepts all outputs before delivery and short-circuits to emergency guidance when needed.
Human-in-the-loop: OCR extractions and high-stakes decisions always request user confirmation before proceeding.
3. LangGraph Node Flow
The graph below shows the end-to-end processing pipeline. All nodes are stateful and share access to the memory store. Branches execute in parallel wherever the tasks are independent.
Figure 2 — LangGraph node-level processing flow
3.1  Node 1 — Input Classifier
The first node receives raw user input and classifies it into one of the following categories:
Text question  (e.g., 'Can I take ibuprofen with milk?')
Prescription image or PDF upload
Lab report upload
Medicine strip / barcode scan
Voice input (converted via Speech-to-Text before classification)
3.2  Node 2 — Memory Retrieval
Before any agent is invoked, the Memory Retrieval node loads all relevant context:
Current medicines and active prescriptions
Known allergies and medical conditions
Previous conversation turns (short-term memory)
Relevant past prescriptions and Q&A (vector similarity search)
Active reminder schedules and adherence history
3.3  Branch A — General Query Agent
Handles free-form medication questions. Execution sequence:
Load user memory and current medicine list
Query Medicine Database API for pharmacological data
Query Chemical Database API for compound information
Run Drug Interaction Checker against user's current medicines
Optional: Internet Search for recent safety advisories
LLM synthesizes a structured, cited answer
3.4  Branch B — Prescription Upload Agent
Processes uploaded prescription images or PDFs through a multi-step pipeline:
OCR engine extracts raw text from the image or PDF
Medicine Extractor identifies drug names, dosages, frequencies, and duration
Confidence Checker flags low-confidence extractions for manual review
User Confirmation step presents extracted data and asks the user to verify before storage
Verified data is stored in the prescription history and triggers the Fan-out Agent
3.5  Branch C — Lab Report Agent
Processes uploaded laboratory reports:
OCR extracts parameter names, values, and units
Units Normalizer converts all values to SI units for consistent comparison
Parameters are stored against the user profile
Parameters are passed to the medication context so the Q&A agent can answer questions like 'Is my kidney function affected by this medicine?'

5. Agent Catalogue
Each agent is a self-contained LangGraph node with its own tools, memory access pattern, and output schema.
5.1  Reminder & Adherence Agent
After a prescription is parsed and confirmed, this agent:
Reads the prescribed schedule (morning / before food / night / after food / SOS)
Asks the user which notification channels they prefer
Creates reminder events in the chosen channels (Telegram, Discord, Email, Calendar, Push)
Marks each reminder as Sent and waits for acknowledgement
On acknowledgement, records Taken; on timeout, asks for reason and records Skipped
Computes weekly and monthly adherence statistics
5.2  Refill Agent
Monitors medicine stock levels and triggers refill workflows:
Calculates estimated days of medicine remaining based on dosage and prescription start date
Sends a refill notification when stock drops below a configurable threshold (default: 5 days)
Presents nearby pharmacy options with availability and price
Optionally initiates an order via integrated pharmacy APIs
5.3  Safety Layer  (Pre-Response Interceptor)
Every response passes through the Safety Layer before delivery. This is not optional — it runs on every single output.
For non-emergency outputs, the Safety Layer enforces these rules:
Never recommend starting, stopping, or changing a prescribed medication
Never diagnose a condition based on symptoms
Always include a disclaimer when discussing dosage or interactions
Flag if a suggested alternative requires a prescription in the user's country

10. Safety & Compliance Considerations
Healthcare applications carry significant ethical and legal responsibilities. The following considerations must be addressed before any public launch.
10.1  Medical Disclaimers
Every response must include a standard disclaimer that the platform is not a substitute for professional medical advice.
Medicine reports must cite their sources explicitly.
The system must never state a diagnosis, even implicitly.
10.2  Data Privacy
Medical data is among the most sensitive categories under every major privacy framework (HIPAA in the US, GDPR in Europe, India's DPDP Act).
All data must be encrypted at rest (AES-256) and in transit (TLS 1.3).
Medical conditions and lab reports must be explicitly opt-in for storage.
Users must be able to export and delete all their data.
AI model providers (OpenAI, Google) must be evaluated for their healthcare data policies before production use.
10.3  Drug Database Accuracy
No drug database is complete. Interaction databases may miss rare combinations.
The system must communicate its data source and last update date for every interaction check.
Users must be advised to verify critical decisions with a pharmacist or clinician.

Emergency Detection Triggers The following symptoms trigger an immediate halt of the normal workflow. The system stops all processing and delivers emergency guidance only:  • Chest pain or tightness • Stroke symptoms (sudden numbness, confusion, vision loss, severe headache) • Signs of anaphylaxis (throat swelling, hives, difficulty breathing) • Suicidal thoughts or expressions of self-harm • Overdose symptoms  In all cases, the system provides emergency contact information appropriate to the user's country and instructs them to seek immediate medical care. No medication advice is given.
